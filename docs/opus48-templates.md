# Opus 4.8 프롬프트·Effort 가이드 (PostOne)

ADU "Claude Opus 4.8 풀세팅 가이드" 요약. 작업 시작 전 effort와 프롬프트 구조를 맞춘다. 핵심: **모델만 바꾸지 말고 effort·프롬프트·작업 방식을 같이 바꾼다.**

## Effort 빠른 선택표

| 작업 | effort |
|---|---|
| 한 문장 다듬기 · 3줄 요약 | Low |
| 분류 · 캡션 다듬기 등 반복 | Medium |
| 일반 실무(기획·분석·요약) | **High** (기본) |
| 코딩 · 에이전트 · 복잡 분석 | **X High** (기본) |
| 결제 플로우 등 대형 전체 구현 | Ultra Code |
| High/X High로 부족할 때만 | Max (테스트용) |

원칙: 결과가 얕으면 프롬프트를 덧붙이지 말고 **effort부터** 올린다.

## 기본 프롬프트 구조

```
목표: [최종 결과]
현재 상황: [상태·배경·문제·관련 파일]
원하는 결과물: [형식·분량·톤·용도]
성공 기준: [좋은 결과의 판단 기준]
작업 방식: 1. 진단 → 2. 계획 제시 → 3. 단계 실행 → 4. 검수
주의사항: [금지 표현·제약]
```

## 규칙

- **"하지 마"보다 "이렇게 해"** — 부정 지시 나열 대신 원하는 방향을 구체적으로.
- **"왜"(목적)를 같이** 준다. 목적을 알면 결과가 좋아진다.
- **응답 길이를 명시**한다(짧게 N줄 / 깊게 원인→구조→방안→주의 / 콘텐츠용).
- 지시는 **모든 섹션에 적용**된다고 명시(첫 항목에만 적용 방지).

## 토큰 절약 스니펫

```
작업 범위를 벗어난 제안은 하지 마세요. 목표 달성에 필요한 내용만 처리해주세요.
먼저 계획만 제시하고, 승인 전까지 실제 수정은 하지 마세요.
가능한 한 적은 파일만 수정하고, 안 건드려도 되는 파일은 두세요.
불필요한 배경 설명은 생략하고, 결정에 필요한 정보와 실행 단계만 주세요.
```

## 코드 리뷰 (→ `review-code` 스킬)

발견(전수 커버리지 + 심각도/확신도) → 필터(실제 리스크만 우선순위). 한 프롬프트에 "중요한 것만"이라 하면 저심각도를 빠뜨리니 **단계를 분리**한다.

## 디자인 (→ `review-design` 스킬)

구현 전 비주얼 방향 옵션(배경 hex·액센트·폰트·레이아웃·근거)을 먼저 뽑게 하고, 선택한 방향만 구현.

## 4.7 → 4.8 전환 체크리스트

**제거**: "계속 진행해 / 멈추지 마 / 끝까지 해 / N번마다 보고 / 게으르게 굴지 마" (4.8에선 과해짐).
**추가**: "범위를 명확히 지켜라 / 불확실하면 추측 말고 확인 필요로 표시 / 수정 전 계획·수정 후 변경사항과 리스크 구분".
